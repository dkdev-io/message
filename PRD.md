## 1. Executive Summary

This document defines the product requirements for an open-source, peer-to-peer (P2P) political messaging application built on Twilio. The platform is designed as a cost-effective alternative to commercial tools such as Hustle and Scale to Win, enabling political campaigns, advocacy organizations, and party committees to manage SMS voter outreach with minimal markup over Twilio costs.

The platform supports two deployment models: a self-hosted open-source installation (SQLite, users purchase Twilio credits directly) and a hosted SaaS version (Supabase/Postgres, Stripe billing). Both versions share the same core codebase.

Key differentiators include LLM-powered campaign script generation, adaptive decision-tree conversations that learn from volunteer interactions, anonymous cross-campaign analytics with AI-driven insights, comprehensive TCPA/FCC compliance built into the core architecture, and native integrations with VAN/EveryAction, NationBuilder, and DDX.

## 2. Problem Statement

Political campaigns and advocacy organizations currently pay significant per-message markups to commercial P2P SMS vendors. These costs can consume a substantial portion of communications budgets, particularly for down-ballot races, local campaigns, and grassroots advocacy groups that operate with limited funding.

Existing solutions are closed-source, preventing customization and creating vendor lock-in. There is no open-source P2P political messaging tool that combines modern LLM capabilities, compliance-first architecture, and direct Twilio integration at cost.

## 3. User Roles & Permissions

| Role | Description | Key Permissions |
|---|---|---|
| Admin | Platform operator (developer/owner) | Full system access, compliance preset overrides, global configuration, user management, system monitoring |
| User | Campaign leadership (campaign manager, field director, etc.) | Create/manage messaging campaigns, upload voter lists, configure scripts, view analytics, manage volunteers, approve LLM-suggested script expansions |
| Volunteer | Field operatives sending messages | Self-register via invite link, send messages per assigned campaign scripts, manage multiple simultaneous conversations, flag unusual voter responses, view personal stats |

A single Volunteer may participate in multiple SMS campaigns from one User, and may also support campaigns from multiple Users. A single User may run multiple messaging campaigns simultaneously.

### 3.1 Volunteer Profiles & Rewards System

Volunteers are persistent entities in the system with their own profiles, not ephemeral per-campaign assignments.

**Volunteer Profile Data Model:**
- Volunteers self-register using an invite link generated by a User
- Profile persists across campaigns and across Users
- Profile stores: total messages sent (lifetime), total conversations completed, total campaigns participated in, average voter response rate, total hours active, date joined
- Volunteers can view a personal dashboard showing their stats across all campaigns

**Volunteer-User Association:**
- VolunteerAssignment record links Volunteer to User and optionally to a specific Campaign
- A Volunteer can have multiple VolunteerAssignment records (many Users, many Campaigns)
- Users can see only the stats for Volunteers on their own campaigns

**Rewards System (Phase 5+):**
- volunteer_achievements table (volunteer_id, achievement_type, earned_at, campaign_id)
- volunteer_stats table that aggregates metrics incrementally
- Achievement thresholds configurable by Admin

## 5. Core Features

### 5.1 Authentication & Payment
- User registers with email/password via Supabase Auth
- Hosted version: Stripe checkout required before campaign creation
- Open source version: User configures their own Twilio credentials
- Volunteer self-registration via invite link with permanent account
- Role-based access control enforced at API and UI layers

### 5.2 Campaign Creation Workflow

**Step 1: Background Document Upload**
- Upload campaign materials for LLM script generation
- Accepts PDF, DOCX, TXT, and URL inputs

**Step 2: Voter List Upload & Validation**
- Required fields: First Name, Last Name, Precinct, Phone Number
- Location/timezone data for TCPA quiet hours enforcement
- Upload accepts .XLS and .XLSX files
- Deduplicate, cross-reference opt-out list, Twilio Lookup API classification
- Summary report with counts by phone type and estimated costs
- Only Mobile and VoIP numbers moved to active campaign

**Step 3: LLM Script Generation**
- SMS-optimized scripts via Anthropic Claude API
- Respect SMS segment length (160 chars)
- Decision-tree response structure
- Scripts are versioned

**Step 4: 10DLC Registration (Automated)**
- Automated TCR submission via Twilio's 10DLC API
- Brand and Campaign registration
- Campaign cannot begin sending until approved

### 5.3 Decision Tree Script Builder & Viewer

**5.3.1 Tree Visualization View (User-Facing)**
- Visual decision tree using React Flow
- Color-coded nodes, edge labels, zoom/pan
- LLM-suggested branches shown as pending approval
- Orphan detection with warning icons

**5.3.2 Volunteer Conversation View**
- Inbox layout with conversation list and chat bubbles
- 1-3 suggested responses from decision tree
- Freeform compose with character counter
- Opt-out indicator and multi-conversation support

**5.3.3 Script Export as PDF**
- Indented outline format with approval checkboxes
- Header with campaign info, footer with signature line

### 5.4 Volunteer Messaging Workflow
- Immediate message sending (no batch queue delay)
- Batch initial sends with rate limiting
- Decision tree conversation flow
- Adaptive script learning with admin approval
- Mid-conversation opt-out handling

### 5.5 Dashboard
- Campaign list with status and real-time metrics
- Volunteer performance tracking
- LLM-powered response analysis with sentiment scoring
- Highlight reel of most positive/negative responses

### 5.6 Analytics & Cross-Campaign Comparison
- Individual campaign analytics
- Anonymous cross-campaign comparison (3-day delay, no geo IDs)
- LLM-powered analytics interface
- Sentiment intensity analysis

## 6. Voter File Integrations

### 6.1 VAN / EveryAction / VoteBuilder
- REST API integration (api.securevan.com/v4)
- HTTP Basic Auth with database modes
- Import/export with canvass response writeback
- Mapping UI for script branches to VAN survey questions

### 6.2 NationBuilder
- JSON:API v2 integration
- OAuth 2.0 with 24-hour token refresh
- Import signups, writeback support levels and tags

### 6.3 DDX (Democratic Data Exchange)
- File-based CSV export/import (no API)
- Requires state_voter_file_id for export eligibility

## 7. Compliance & Legal Requirements

### 7.1 TCPA Federal Baseline
- Honor all opt-out requests immediately
- Opt-out records retained minimum 4 years

### 7.2 Quiet Hours — System-Enforced

| Time Window | Rule | Override |
|---|---|---|
| Weekdays (Mon-Thu) | No messages before 8:00 AM or after 9:00 PM local time | None — hard limit |
| Friday | No messages after 8:00 PM local time | Admin approval required |
| Saturday | No messages before 10:00 AM local time | Admin approval required |
| Sunday | No messages before 12:00 PM local time | Admin approval required |
| Sunday evening | No messages after 9:00 PM local time | None — hard limit |

State-level quiet hours table for FL, TX, OK, etc.

### 7.3 Global Opt-Out List
- Cross-campaign, cross-user opt-out list
- Phone numbers stored as SHA-256 hashes

### 7.4 10DLC Compliance
- Registration mandatory before sending
- Automated via Twilio API

### 7.5 Record Retention
- Minimum 4 years for all compliance data

## 10. Non-Functional Requirements

- Message send latency: < 2 seconds
- LLM reply categorization: < 3 seconds
- Batch processing: 10,000+ numbers with progress indicator
- Dashboard: WebSocket or SSE for real-time updates
- All credentials encrypted at rest (AES-256)
- HTTPS/TLS 1.3
- Supabase Row Level Security for multi-tenant isolation

## 11. Implementation Phases

1. **Phase 1: Core MVP** — Auth, campaign creation, voter list upload, basic script editor, volunteer assignment, P2P messaging, conversation inbox, STOP handling, quiet hours, dashboard
2. **Phase 2: LLM Integration** — Script generation, decision tree UI, reply categorization, adaptive learning
3. **Phase 3: Analytics** — Campaign analytics, cross-campaign comparison, LLM analytics chat, PDF export
4. **Phase 4: Integrations** — VAN, NationBuilder, DDX, 10DLC automation, Stripe, state quiet hours
5. **Phase 5: Mobile & Scale** — Mobile app, volunteer rewards, performance optimization
